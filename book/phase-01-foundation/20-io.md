## Story

Bạn cần đọc một file text chứa danh sách sản phẩm, mỗi dòng một sản phẩm. Việc tưởng đơn giản
này trong `java.io` lại cần **ba lớp lồng nhau**:

```java
BufferedReader reader = new BufferedReader(new FileReader(file));
```

`FileReader` bọc trong `BufferedReader`. Tại sao không đọc thẳng bằng `FileReader`? Và tại
sao có `FileReader` (đọc ký tự) lại còn có `FileInputStream` (đọc byte) — không phải chỉ cần
một cách đọc file là đủ sao? Chapter này giải thích lý do đằng sau cấu trúc "lồng nhau" tưởng
như rườm rà này — mỗi lớp bọc thêm giải quyết đúng một vấn đề cụ thể, không phải sự phức tạp
ngẫu nhiên.

## Interview Question (Central)

## Problem

Dữ liệu bên ngoài JVM (file, mạng, bàn phím) đến dưới dạng **byte thô** — máy tính không có
khái niệm "ký tự" ở tầng phần cứng. Nhưng phần lớn dữ liệu con người làm việc là **văn bản**
(ký tự, cần một bảng mã như UTF-8 để biết byte nào ghép thành ký tự nào). Nếu chỉ có một API
đọc/ghi byte thô duy nhất, mọi thao tác với văn bản sẽ phải tự tay xử lý encoding — dễ sai,
lặp lại code ở khắp nơi.

## Concept

`java.io` chia làm hai "họ" song song, giải quyết hai bài toán khác nhau:

- **`InputStream`/`OutputStream`** — đọc/ghi **byte thô**, không quan tâm ý nghĩa (dùng cho
  file nhị phân: ảnh, file nén, dữ liệu mạng thô).
- **`Reader`/`Writer`** — đọc/ghi **ký tự** (tự động xử lý encoding, ví dụ UTF-8, quy đổi
  giữa byte và ký tự) — dùng cho văn bản.

`FileReader`, `FileWriter`, `BufferedReader`... đều thuộc họ `Reader`/`Writer`. Việc "lồng"
nhiều lớp (như `BufferedReader` bọc `FileReader`) theo đúng **Decorator Pattern**: mỗi lớp
bọc thêm một khả năng, không phải viết lại từ đầu.

## Why?

Nếu không tách byte và ký tự: mọi lần đọc văn bản đều phải tự chuyển đổi encoding thủ công —
dễ viết sai (một lỗi encoding rất phổ biến: đọc file UTF-8 bằng encoding mặc định của hệ điều
hành khác, ra ký tự lỗi (mojibake) — chính là hệ quả của việc lẫn lộn giữa byte thô và ký tự).

Nếu không có buffer: mỗi lời gọi `read()` (đọc một byte/ký tự) tương ứng với **một lời gọi hệ
thống** (system call) xuống hệ điều hành để lấy dữ liệu từ đĩa — cực kỳ tốn kém nếu gọi hàng
trăm nghìn lần cho một file lớn (đọc từng ký tự một). `BufferedReader` đọc một khối lớn dữ liệu
vào bộ nhớ đệm (buffer) trong **một** lời gọi hệ thống, rồi phục vụ các lời gọi `read()` tiếp
theo **từ bộ nhớ trong JVM** — nhanh hơn rất nhiều.

## How?

```
Dữ liệu thô (đĩa/mạng)
        │
        ▼
InputStream/Reader gốc (FileInputStream/FileReader)
   — đọc TRỰC TIẾP từ hệ điều hành, MỖI LỜI GỌI read() = MỘT SYSTEM CALL
        │
        │  bọc thêm (Decorator Pattern)
        ▼
BufferedInputStream/BufferedReader
   — đọc một KHỐI LỚN vào bộ nhớ đệm trong MỘT system call,
     các read() tiếp theo phục vụ từ bộ nhớ JVM (nhanh hơn nhiều)
```

## Example

Ghi và đọc một file văn bản bằng try-with-resources (Java 7+, tự động đóng stream dù có lỗi
hay không — không cần `finally` thủ công như Chapter 15):

```java
import java.io.*;

public class IoDemo {
    public static void main(String[] args) throws IOException {
        File file = new File("product.txt");
        try (BufferedWriter writer = new BufferedWriter(new FileWriter(file))) {
            writer.write("Ao thun,199000");
            writer.newLine();
            writer.write("Quan jean,399000");
        }

        try (BufferedReader reader = new BufferedReader(new FileReader(file))) {
            String line;
            while ((line = reader.readLine()) != null) {
                System.out.println("Doc: " + line);
            }
        }
        file.delete();
    }
}
```

Kết quả thật:

```
Doc: Ao thun,199000
Doc: Quan jean,399000
```

**Đo chênh lệch hiệu năng có/không có buffer** — đọc cùng một file 50.000 dòng, từng ký tự
một, có và không có `BufferedReader`:

```java
File file = new File("big.txt");
// ghi 50.000 dòng vào file (bỏ qua ở đây, xem đầy đủ ở Exercises)

long start1 = System.nanoTime();
try (FileReader r = new FileReader(file)) {
    while (r.read() != -1) { /* đếm */ }
}
long t1 = (System.nanoTime() - start1) / 1_000_000;

long start2 = System.nanoTime();
try (BufferedReader r = new BufferedReader(new FileReader(file))) {
    while (r.read() != -1) { /* đếm */ }
}
long t2 = (System.nanoTime() - start2) / 1_000_000;
```

Kết quả thật (JDK 17, file ~689.000 ký tự):

```
Khong buffer (doc tung byte truc tiep tu FileReader): 22 ms, 688890 ky tu
Co BufferedReader: 4 ms, 688890 ky tu
```

Chênh lệch khoảng **5.5 lần** — bằng chứng thực nghiệm trực tiếp cho lý do ở phần Why?.

## Deep Dive

**Try-with-resources hoạt động như thế nào bên dưới?** Cú pháp `try (Resource r = ...) { }`
(Java 7+) yêu cầu `Resource` implement `AutoCloseable` (đã nhắc thoáng qua ở
[Chapter 09](09-object.md) khi bàn về thay thế `finalize()`) — trình biên dịch **tự động** sinh
ra một khối tương đương `try-finally` với `r.close()` được gọi trong `finally`, đảm bảo stream
luôn được đóng dù có exception xảy ra hay không, mà không cần tự viết `finally` thủ công như
trước Java 7 (và tránh được đúng cạm bẫy `finally`/`return` đã học ở
[Chapter 15](15-exception.md), vì `close()` sinh tự động không có `return` gây nuốt exception).

## Engineering Insight

**Vì sao "buffer" lại hiệu quả đến vậy — bản chất kỹ thuật đằng sau là gì?** Vì system call
(yêu cầu hệ điều hành thực hiện một thao tác, như đọc dữ liệu từ đĩa) có chi phí **cố định rất
lớn** so với việc chỉ đọc dữ liệu đã có sẵn trong bộ nhớ JVM (chuyển ngữ cảnh sang kernel, chờ
kernel xử lý, chuyển ngữ cảnh trở lại — tốn hàng nghìn chu kỳ CPU cho MỖI lần gọi, bất kể đọc
1 byte hay 8000 byte). Buffer khai thác đúng nguyên lý: "trả giá system call **một lần** cho
một khối lớn, rồi phục vụ nhiều yêu cầu nhỏ từ bộ nhớ đã có sẵn" — cùng triết lý tối ưu đã gặp
ở nhiều chapter trước theo những cách khác nhau: JIT Compiler (Chapter 06) trả giá biên dịch
một lần, dùng lại nhiều lần; String Pool (Chapter 10) trả giá tạo object một lần, dùng chung
nhiều biến.

## Myth vs Reality

- **Myth:** "`FileInputStream` và `FileReader` về cơ bản làm cùng một việc, chỉ khác tên."
  **Reality:** `FileInputStream` đọc byte thô; `FileReader` đọc ký tự (tự xử lý encoding) —
  dùng sai loại cho văn bản có thể gây lỗi hiển thị ký tự (đặc biệt với tiếng Việt có dấu, nếu
  encoding không khớp).

- **Myth:** "Bọc thêm `BufferedReader` chỉ là thói quen tốt, không thực sự cần thiết trừ file
  cực lớn."
  **Reality:** Ngay cả với file vừa phải (Example: chưa tới 700.000 ký tự), chênh lệch đã hơn
  5 lần — buffer gần như luôn nên dùng mặc định khi đọc/ghi file hoặc stream theo từng
  đơn vị nhỏ (ký tự, dòng), trừ khi chính API bạn dùng đã tự buffer sẵn bên trong.

## Common Mistakes

- **Quên đóng stream** (trước Java 7, quên viết `finally { close(); }`) — gây rò rỉ file
  handle, một dạng "leak" khác với memory leak (Chapter 07) nhưng cùng bản chất: giữ tài
  nguyên không giải phóng. Try-with-resources (Java 7+) gần như loại bỏ hoàn toàn lỗi này nếu
  dùng đúng.
- **Đọc file không qua buffer trong vòng lặp lớn** — đúng lỗi hiệu năng đã đo ở Example.
- **Lẫn lộn byte và ký tự**, đặc biệt khi làm việc với encoding không phải mặc định của hệ
  thống — luôn chỉ định encoding tường minh (`new InputStreamReader(in, StandardCharsets.UTF_8)`)
  thay vì dựa vào encoding mặc định của máy đang chạy, vốn có thể khác nhau giữa các môi
  trường (dev, CI, production).

## Best Practices

- Luôn dùng try-with-resources cho mọi stream — không viết `close()` thủ công trong `finally`.
- Luôn bọc `Buffered...` khi đọc/ghi theo đơn vị nhỏ (ký tự, dòng) lặp lại nhiều lần.
- Luôn chỉ định encoding tường minh khi làm việc với `Reader`/`Writer`, không dựa vào mặc định
  của hệ điều hành.

## Production Notes

**Vấn đề:** dịch vụ xử lý file log/export dữ liệu lớn (hàng trăm MB) chạy rất chậm, chiếm
nhiều CPU hơn dự kiến so với khối lượng dữ liệu thực tế.

- **Triệu chứng:** thời gian xử lý tăng gần như tuyến tính (hoặc tệ hơn) theo kích thước file,
  CPU usage cao bất thường trong lúc I/O (thường ngược lại — I/O thường ở trạng thái chờ, ít
  tốn CPU).
- **Root cause:** code đọc/ghi trực tiếp qua `FileReader`/`FileInputStream` không qua buffer,
  từng ký tự/byte một trong vòng lặp — mỗi lần đọc kích hoạt một system call (xem Engineering
  Insight), tạo áp lực lớn lên cả CPU (chuyển ngữ cảnh liên tục) lẫn thời gian chờ I/O.
- **Debug:** review code đọc/ghi file, tìm chỗ dùng trực tiếp lớp gốc
  (`FileReader`/`FileInputStream`/`FileWriter`/`FileOutputStream`) mà không có
  `Buffered...` bọc quanh.
- **Solution:** thêm `BufferedReader`/`BufferedInputStream`/`BufferedWriter`/
  `BufferedOutputStream` bọc quanh, hoặc dùng các API cấp cao hơn (như NIO.2's
  `Files.readAllLines()`, Chapter 21) đã tự buffer sẵn bên trong.
- **Prevention:** đưa "mọi truy cập file phải qua Buffered stream" vào coding standard/code
  review checklist; với xử lý dữ liệu khối lượng lớn, cân nhắc NIO (Chapter 21) ngay từ đầu.

## Debug Checklist

- [ ] Xử lý file chậm bất thường, CPU cao khi I/O? → tìm chỗ đọc/ghi trực tiếp không qua
      `Buffered...`.
- [ ] Ký tự tiếng Việt/ký tự đặc biệt hiển thị sai (mojibake)? → kiểm tra encoding có được chỉ
      định tường minh không, có khớp giữa lúc ghi và lúc đọc không.
- [ ] Nghi ngờ rò rỉ file handle (lỗi "too many open files" sau một thời gian chạy)? → tìm
      những nơi mở stream không dùng try-with-resources, có khả năng quên đóng khi có
      exception.

## Source Code Walkthrough

`FileInputStream.read()` (byte đơn) cuối cùng gọi xuống một method `native` (hiện thực bằng
C, tương tự các method `native` của `Object` đã gặp ở [Chapter 09](09-object.md)) — thực hiện
system call thật xuống hệ điều hành. `BufferedInputStream` không override cơ chế đó, mà **bọc
quanh** một `InputStream` khác (đúng Decorator Pattern ở phần Concept): nó tự quản lý một mảng
byte nội bộ, chỉ gọi xuống `InputStream` gốc khi buffer đã cạn, cần nạp thêm một khối lớn mới —
đúng cơ chế "trả giá system call một lần cho nhiều lần đọc nhỏ" đã phân tích ở Engineering
Insight.

## Summary

`java.io` chia thành họ đọc/ghi **byte** (`InputStream`/`OutputStream`, cho dữ liệu nhị phân)
và họ đọc/ghi **ký tự** (`Reader`/`Writer`, tự xử lý encoding, cho văn bản) — dùng đúng loại
tránh lỗi encoding. Các lớp `Buffered...` bọc thêm bộ nhớ đệm theo đúng Decorator Pattern,
giảm số lượng system call từ hàng nghìn xuống một vài lần — chênh lệch hiệu năng đo được thực
tế hơn 5 lần chỉ với một file vừa phải. Try-with-resources (Java 7+) tự động đóng stream đúng
cách, loại bỏ lớp lỗi "quên đóng tài nguyên" phổ biến trước đó.

## Cheat Sheet

| Nhóm | Đọc/ghi gì | Ví dụ class |
| --- | --- | --- |
| `InputStream`/`OutputStream` | Byte thô | `FileInputStream`, `FileOutputStream` |
| `Reader`/`Writer` | Ký tự (tự xử lý encoding) | `FileReader`, `FileWriter` |
| `Buffered...` | Bọc thêm bộ nhớ đệm | `BufferedReader`, `BufferedInputStream` |

| Nên dùng | Thay vì |
| --- | --- |
| `try (Resource r = ...) { }` | `try { } finally { r.close(); }` thủ công |
| `new InputStreamReader(in, StandardCharsets.UTF_8)` | Dựa vào encoding mặc định hệ thống |
| `BufferedReader`/`BufferedWriter` khi đọc/ghi lặp nhiều lần | Đọc/ghi trực tiếp không buffer |
