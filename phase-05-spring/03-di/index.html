    Dependency Injection (DI)¶

    Story¶

    Xem lại: Chapter 02 — IoC là nguyên lý "quyền tạo dependency thuộc về bên ngoài". Dependency Injection (DI) là cách Spring hiện thực hoá nguyên lý đó — nhưng có tới 3 cách "tiêm" khác nhau (constructor, field, setter). Chúng có thực sự tương đương nhau không?

    Thử tạo hai bean phụ thuộc vòng tròn lẫn nhau (A cần B, B cần A) — trước hết bằng constructor injection:

    Constructor injection circular dependency -> UnsatisfiedDependencyException
Message rut gon: ... Requested bean is currently in creation: Is there an unresolvable circular reference?

    Container từ chối khởi động. Thử lại với field injection (@Autowired trên field thay vì constructor):

    Field injection: KHOI DONG THANH CONG du co circular dependency
a.b == b: true
b.a == a: true

    Container khởi động bình thường, dependency vòng tròn vẫn được nối đúng! Hai cách injection này rõ ràng không hề tương đương.

    Interview Question (Central)¶

    Có bao nhiêu cách Dependency Injection trong Spring? Vì sao constructor injection được khuyến nghị hơn field injection?

    Objectives¶

    • [ ] Phân biệt 3 cách DI: constructor, field, setter injection
    • [ ] Tự tay chứng minh bằng thực nghiệm: constructor injection phát hiện circular dependency ngay lúc khởi động, field injection thì không
    • [ ] Giải thích cơ chế "early reference" cho phép field injection giải quyết được circular dependency mà constructor injection không làm được

    Prerequisites¶

    • Chapter 02 — hiểu IoC, container tạo và tiêm dependency thay vì để bean tự tạo.

    Used Later¶

    • Chapter 04 (Bean) — cách container quyết định thứ tự tạo bean dựa trên dependency graph.
    • Chapter 12 (AOP), Chapter 13 (Transaction) — cả hai đều dựa trên cơ chế proxy, tương tác trực tiếp với cách bean được tiêm và gọi lẫn nhau.

    Problem¶

    Circular dependency (A cần B, B cần A) là một tình huống thực tế có thể xảy ra khi thiết kế không cẩn thận. Với constructor injection, để tạo A hoàn chỉnh cần B đã hoàn chỉnh, nhưng để tạo B hoàn chỉnh lại cần A đã hoàn chỉnh — một vòng lặp logic không có điểm bắt đầu. Ba cách DI khác nhau (constructor/field/setter) xử lý tình huống này hoàn toàn khác nhau, và người mới học thường không biết rằng chọn sai cách DI có thể che giấu một lỗi thiết kế nghiêm trọng thay vì để nó lộ ra sớm.

    Concept¶

    Dependency Injection là cách IoC container "tiêm" (cung cấp) dependency vào một bean, thay vì để bean tự tạo. Có 3 cách: constructor injection (tiêm qua tham số constructor, dependency là final, bất biến sau khi tạo), field injection (tiêm trực tiếp vào field qua Reflection, thường dùng @Autowired trên field), setter injection (tiêm qua một phương thức setter, cho phép thay đổi dependency sau khi bean đã tạo).

    Why?¶

    Constructor injection buộc một bean phải hoàn chỉnh ngay khi được tạo ra (mọi dependency bắt buộc đã sẵn sàng trước khi constructor chạy xong) — điều này khiến circular dependency trở thành một tình huống không thể giải quyết được về mặt logic thuần tuý (A cần B hoàn chỉnh, B cần A hoàn chỉnh, không ai có thể "đi trước"), nên Spring buộc phải báo lỗi ngay khi phát hiện. Field/ setter injection cho phép tạo object trước (constructor không tham số, hoặc constructor không cần dependency), rồi điền dependency vào sau — Spring tận dụng khoảng trống này bằng cách tạo một "early reference" (tham chiếu tới object đã tồn tại nhưng dependency bên trong nó CHƯA hoàn tất), đưa reference đó cho bên đang chờ để tiếp tục hoàn tất, rồi hoàn thiện lại sau. Đây chính là lý do field injection "giải quyết" được circular dependency trong khi constructor injection thì không.

    How?¶

    // Constructor injection (KHUYEN NGHI)
@Service
class OrderService {
    private final PaymentService paymentService; // co the la 'final'
    OrderService(PaymentService paymentService) { this.paymentService = paymentService; }
}

// Field injection (khong khuyen nghi cho production code)
@Service
class OrderService {
    @Autowired
    private PaymentService paymentService; // KHONG the la 'final'
}

// Setter injection (hiem dung, chi khi that su can doi dependency sau khi tao)
@Service
class OrderService {
    private PaymentService paymentService;
    @Autowired
    void setPaymentService(PaymentService paymentService) { this.paymentService = paymentService; }
}

    Visualization¶

    Constructor injection + circular dependency:

  Tao A -> can B hoan chinh -> Tao B -> can A hoan chinh -> Tao A -> ... (VONG LAP, khong loi ra)
  => Spring PHAT HIEN va BAO LOI ngay: "Is there an unresolvable circular reference?"

Field injection + circular dependency:

  Tao A (constructor KHONG can B) -> A ton tai (nhung field b con trong)
       -> Tao B (constructor KHONG can A) -> B can dien field a
             -> Spring dua "early reference" cua A (chua hoan tat) cho B
       -> B hoan tat (field a = A)
  -> Quay lai dien field b cua A = B (da hoan tat)
  => CA HAI deu hoan tat, KHONG loi

    Example¶

    static class AConstructor { AConstructor(BConstructor b) {} }
static class BConstructor { BConstructor(AConstructor a) {} }

@Configuration
static class ConfigConstructor {
    @Bean AConstructor a(BConstructor b) { return new AConstructor(b); }
    @Bean BConstructor b(AConstructor a) { return new BConstructor(a); }
}
// new AnnotationConfigApplicationContext(ConfigConstructor.class) -> LOI

@Component
static class AField { @Autowired BField b; }
@Component
static class BField { @Autowired AField a; }
// new AnnotationConfigApplicationContext(AField.class, BField.class) -> THANH CONG

    Kết quả thật (Spring Framework 6.1.13):

    Constructor injection circular dependency -> UnsatisfiedDependencyException
Message rut gon: ... Requested bean is currently in creation: Is there an unresolvable circular reference?

Field injection: KHOI DONG THANH CONG du co circular dependency
a.b == b: true
b.a == a: true

    Với constructor injection, container ném UnsatisfiedDependencyException ngay khi khởi động — không thể tạo được A vì cần B hoàn chỉnh, không thể tạo B vì cần A hoàn chỉnh. Với field injection, container khởi động thành công, và a.b == b, b.a == a đều true — cả hai bean được nối đúng lẫn nhau, không hề có lỗi nào xuất hiện.

    Deep Dive¶

    Cơ chế "early reference" cho phép field injection giải quyết circular dependency hoạt động chi tiết ra sao? Spring container dùng một kỹ thuật gọi là "three-level cache" (ba tầng cache) khi tạo bean singleton: khi bắt đầu tạo A (constructor không cần B, A được tạo xong ở dạng "thô", field b chưa được điền), Spring đăng ký một factory tạo early reference cho A vào cache tầng 3, trước khi thực sự điền các dependency của A. Khi B (đang được tạo, cần A) hỏi container "cho tôi A", container thấy A đang trong quá trình tạo (currently in creation), nó tra cache tầng 3, gọi factory đó để lấy "early reference" của A (dù A chưa hoàn tất), trả về cho B. B hoàn tất với reference "thô" của A này. Sau đó, A tiếp tục hoàn tất bằng cách điền field b = B (đã hoàn tất). Cơ chế này chỉ khả thi vì field injection cho phép object tồn tại (dù chưa hoàn chỉnh) trước khi dependency được điền — constructor injection không có khoảng trống này vì object không tồn tại cho tới khi constructor (cần toàn bộ dependency) chạy xong.

    Engineering Insight¶

    Nếu field injection "giải quyết được" circular dependency, tại sao vẫn khuyến nghị constructor injection? Đây chính là điểm mấu chốt hay bị hiểu sai: field injection không "giải quyết" vấn đề circular dependency — nó che giấu một lỗi thiết kế thực sự. Circular dependency giữa hai bean gần như luôn là dấu hiệu của trách nhiệm bị phân chia sai (A và B nên được gộp lại, hoặc cần tách ra một bean thứ ba chứa phần logic dùng chung). Constructor injection "ép" lỗi thiết kế này phải lộ diện ngay lúc khởi động — thời điểm rẻ nhất, an toàn nhất để phát hiện và sửa. Field injection cho phép ứng dụng "chạy được" dù thiết kế có vấn đề, khiến lỗi tiềm ẩn này có thể tồn tại âm thầm rất lâu trong codebase, khó phát hiện qua code review thông thường (vì không có exception nào cả). Đây, cùng với việc constructor injection cho phép field là final (bất biến, dễ test với new OrderService(mockPaymentService) không cần Reflection), là lý do chính khiến constructor injection được khuyến nghị mạnh mẽ trong tài liệu chính thức của Spring.

    Historical Note¶

    Circular dependency resolution qua field/setter injection (three-level cache) đã tồn tại từ các phiên bản Spring rất sớm. Việc khuyến nghị mạnh constructor injection là xu hướng rõ rệt hơn trong tài liệu và cộng đồng Spring những năm gần đây — phản ánh kinh nghiệm thực tế tích luỹ: field injection dễ viết hơn lúc đầu (không cần constructor dài dòng với nhiều tham số), nhưng lại là nguồn gốc phổ biến của thiết kế "God class" (một bean phụ thuộc quá nhiều thứ, dễ nhận ra qua constructor quá dài — một tín hiệu cảnh báo tự nhiên mà field injection không có).

    Myth vs Reality¶

    • Myth: "Field injection và constructor injection hoàn toàn tương đương, chỉ khác cú pháp." Reality: Đã chứng minh bằng thực nghiệm — chúng khác nhau căn bản trong việc phát hiện circular dependency, và trong khả năng dùng final field (constructor injection có, field injection không).

    • Myth: "Circular dependency chạy được (qua field injection) nghĩa là thiết kế ổn, không cần sửa." Reality: Xem Engineering Insight — nó chỉ là "chạy được", không có nghĩa là thiết kế đúng. Gần như luôn nên coi circular dependency là tín hiệu cần tái cấu trúc.

    Common Mistakes¶

    • Dùng field injection cho mọi dependency vì "ngắn gọn hơn" — mất khả năng dùng final, mất khả năng phát hiện circular dependency sớm, và khó test hơn (cần Reflection hoặc @InjectMocks của Mockito thay vì new trực tiếp).
    • Coi việc field injection "giải quyết" được circular dependency là một giải pháp hợp lệ — nên coi đó là dấu hiệu cần tái cấu trúc thiết kế, không phải một tính năng để tận dụng.
    • Viết constructor quá dài với hàng chục tham số — dấu hiệu bean đang gánh quá nhiều trách nhiệm, nên tách nhỏ.

    Best Practices¶

    • Luôn ưu tiên constructor injection cho dependency bắt buộc (required dependency).
    • Coi lỗi UnsatisfiedDependencyException liên quan tới circular reference là tín hiệu cần tái cấu trúc (tách bean, hoặc gộp lại), không tìm cách "vượt qua" bằng field injection.
    • Dùng setter injection chỉ khi dependency thực sự tuỳ chọn (optional) hoặc cần thay đổi được sau khi bean đã tạo — trường hợp hiếm gặp trong thực tế.
    • Với Lombok, @RequiredArgsConstructor trên các field private final là cách phổ biến để có constructor injection mà không cần viết constructor thủ công.

    Debug Checklist¶

    • [ ] UnsatisfiedDependencyException với thông điệp "Is there an unresolvable circular reference?"? → xác nhận đang dùng constructor injection giữa hai (hoặc nhiều) bean phụ thuộc vòng tròn — nên tái cấu trúc thay vì đổi sang field injection để "né" lỗi.
    • [ ] Field injection dependency là null dù bean đã được tạo? → kiểm tra class có được Spring quản lý (đúng là một bean, không phải new thủ công) không — field injection chỉ hoạt động trên object do container tạo.

    Summary¶

    Dependency Injection có 3 cách: constructor (khuyến nghị, bắt buộc dependency hoàn chỉnh trước khi object tồn tại), field (tiện nhưng mất final, khó test), setter (hiếm dùng, cho dependency tuỳ chọn). Đã chứng minh bằng thực nghiệm: constructor injection phát hiện circular dependency ngay lúc khởi động (UnsatisfiedDependencyException), trong khi field injection "giải quyết" được nhờ cơ chế early reference (three-level cache) — nhưng đây là che giấu lỗi thiết kế chứ không phải giải pháp thực sự. Circular dependency gần như luôn là dấu hiệu cần tái cấu trúc, và constructor injection giúp phát hiện điều đó sớm nhất có thể.

    Interview Questions¶

    Exercises¶

    • [ ] Chạy lại CircularDependencyTest ở trên, xác nhận constructor injection lỗi còn field injection thành công đúng như mô tả.
    • [ ] Tái cấu trúc ví dụ circular dependency (A, B) thành thiết kế không còn vòng tròn — ví dụ tách phần logic dùng chung ra một bean thứ ba C.
    • [ ] Viết một bean dùng setter injection cho một dependency tuỳ chọn, minh hoạ trường hợp hợp lệ để dùng setter injection thay vì constructor.
